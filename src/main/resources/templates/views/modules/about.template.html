About this project

Our approach

Our goal was to create an application using data from Open FDA that demonstrated various ways of consumption. This application shows 4 different approaches of consuming and presenting said data.

Initial stage

When starting the project during our whiteboarding sessions, our initial concern was formulating an idea to proceed with the various types of data provided. We saw that the Open FDA site already provided graphical information about some sample queries so we understood that we would need to go beyond simple graphing. One part that jumped out was the statistic on one of the Open FDA graphs about recalls with the word "ice cream" in the description. So as a first go, we came up with the idea that it would be very informative to generate headlines using the data from Open FDA in a human readable format as opposed to a graph.

This idea worked well, and some of us being Bill Nye fans, we decided to name the site "Consider the following", based on a segment of his "Bill Nye the Science Guy" show. His segment presented viewers with cool facts about various subjects so we decided to present cool automatically generated statistics derived from the Open FDA data. With our main idea in place, we decided to work with all three types of datasets (food, medicines, and medical devices). Our goal was to show three types of data interaction based on the amount and usability of the data Open FDA provides.

Through some brainstorming, we came up with the following:

  1. For Food, we show the site user recalls by state
  2. For Medicines, we allow the user to search for events related to a certain medicine
  3. For Medical Devices, we show the user a pre-defined search and allow them to view details.

Food

The goal of this page is to show food recalls that affect a given state. In order to make it more usable, the application pre-selects the current user's state based on geolocation information. The application also highlights information about the recall and populates a US map based on other areas affected by the same recall. If the user needs more information about the manufacturer, the application provides a link to Google search results based on the manufacturer's name and address.

Medicines

The medicines page allows the user to search for one or many medcines to generate a graph of adverse events reported in which the patient was known to be taking those medicines. The application queries Open FDA for counts of various serious adverse events including deaths, hopitalizations, and life threatening events. Using this data, the application generates a stacked bar chart using the C3 graphing library.

Medical Devices

The data the Open FDA API provides for medical devices doesn't lend itself to the general public. Due to this, the application keeps this screen simple and renders the results of a pre-defined query to a list of results. The user can then see the details for each result to get information about a potential device recall including manufacturer information. The manufacturer information is also linked to Google and their telephone number is optimized for quick communication with the manufacturer (tap it on a mobile phone to dial directly).

Our technology

This application is built entirely on an open-source application stack. We use the very popular Spring framework to create a Java Web Application which can run as a standalone application or inside a Java container. For the front-end we use AngularJS and PureCSS as our JavaScript MVC framework and CSS framework respectively.

Running on an open-source stack we were able to build this application from the bottom up in a matter of 2 weeks. A more detailed explanation of our technology stack can be found at our Github repository

Our processes

One of our goals was to manage the creation of this prototype as an Agile project. Therefore, our processes were at a minimum and evolved from day one. We used Github's issue tracking mechanism to track our user stories, tasks, and bugs. We created sprints by grouping issues into milestones. This break up allowed us to have focused tasks during our short sprint intervals. Each person in our team had defined roles but it did not mean we didn't help each other when necessary.

Bashing bugs!

Various people from our team took an active part in finding and documenting bugs as Github issues during our development. On Monday June 29, 2015 we also held a Bug Bash session to find as many issues as we could on the application in a two hour period. The testing performed by our team helped polish the application and led to various design changes ranging from the UI to our backend (we had to implement smart caching logic).

Getting our code out the door

Deployments to our production environment were controlled through Github pull requests that needed approval by our Product Owner. All the builds and deployments were automatically done by TravisCI. You can get more information about the technical aspects of our builds and deployments by visiting our CI/CD documentation on Github.
