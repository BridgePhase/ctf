About this project

Our approach

Our goal was to create an application using data from openFDA that demonstrated various ways of consumption. This application shows 4 different approaches of consuming and presenting said data.

Initial stage

During our initial whiteboarding sessions, we brainstormed what could be done with the various types of data provided. We saw that the openFDA site already provided graphical information using some sample queries, so we knew we would need to go beyond simple graphing. One part that jumped out was a statistic on a graph about recalls with the word "ice cream" in the description. We then came up with the idea that it would be very informative to generate headlines using the data from openFDA in a human readable format.

This idea worked well, and some of us being Bill Nye fans, decided to name the site "Consider the following" which is based on a segment of his Bill Nye the Science Guy show. His segment presented viewers with cool facts about various subjects, so our headlines are meant to present cool automatically generated statistics derived from the openFDA data. With the first method of data consumption in place, we decided to work with all three types of datasets (food, medicines, and medical devices) in order to demonstrate three additional types of data interaction based on the amount and usability of the data openFDA provides.

Through some brainstorming, we came up with the following:

  1. For Food, we show recalls by state
  2. For Medicines, we provide a search for events related to a certain medicines
  3. For Medical Devices, we show a pre-defined search and the ability to view details.
  4. For the homepage, we generate human readable headlines based on the statistics retrieved from openFDA

Food

The goal of this page is to show food recalls that affect a given state. In order to make it more usable, the application pre-selects the current user's state based on geolocation information. The application also highlights information about the recall and populates a US map based on other areas affected by the same recall. If the user needs more information about the manufacturer, the application provides a link to Google search results based on the manufacturer's name and address.

Medicines

The medicines page allows the user to search for one or many medicines. Based on that search, the application queries openFDA for counts of various serious adverse events including deaths, hospitalizations, and life threatening events. Using this data, the application generates a stacked bar chart grouped by patient age using the C3 graphing library.

Medical Devices

The data the openFDA API provides for medical devices does not lend itself to the general public. Due to this, the application keeps this screen simple and renders the results of a pre-defined query to a list of results. The user can then see the details for each result to get information about a potential device recall including manufacturer information. The manufacturer information is also linked to Google and their telephone number is optimized for quick communication with the manufacturer (tap it on a mobile phone to dial directly).

Headlines

We implemented human readable headlines for the homepage for two reasons. First, it grabs the user's attention when navigating to the homepage by presenting interesting facts dealing with data from openFDA. Second, it showcases a different manner of interacting with data by providing more than graphs and tables. The headlines are generated from count queries using pre-defined criteria, one of the future goals is to make this criteria more dynamic.

Our technology

This application is built entirely on an open-source application stack. We use the Spring framework to create a Java Web Application which can run as a standalone application or inside a Java container. For the front-end we use AngularJS and PureCSS as our JavaScript MVC framework and CSS framework respectively.

Running on an open-source stack we were able to build this application from the bottom up in a matter of 2 weeks. A more detailed explanation of our technology stack can be found at our Github repository

Our processes

One of our goals was to manage the creation of this prototype as an Agile project. Therefore, our processes were at a minimum and evolved from day one. Grouping issues into milestones created our sprints, and we used Github's issue tracking mechanism to track our user stories, tasks, and bugs. This break up allowed us to have focused tasks during our short sprint intervals. Each person in our team had defined roles but it did not mean we did not help each other when necessary.

Bashing bugs!

Various people from our team took an active part in finding and documenting bugs as Github issues during development. On Monday, June 29, 2015 we also held a "Bug Bash"" session to find as many issues as we could in the application within a two hour period. The testing performed by our team helped polish the application and led to various design changes ranging from the UI to our backend ( smart caching logic for example).

Getting our code out the door

All the builds and deployments were automatically done by TravisCI. Deployments to our production environment were controlled through Github, and pull requests to `master` first required approval by our Product Owner.
